### Engineer-reported lifecycle decisions

> Retry is consuming: `Run<Failed>::retry(self) -> Run<Pending>`, matching the existing sealed transition style. Lineage lives directly on `Run<S>` so it survives `start`, `complete`, and `fail`. `derive_run_id(intent, claims, retried_from, retry_index)` now hashes serialized lineage plus big-endian retry index after the existing run hash inputs.

Watermaster read: Codex chose the four design calls the Brief flagged as Codex's call exactly the way I would have:

1. **Consuming retry**: matches the sealed-transition discipline; failed run "spent" by retry; chains of retries formed by retrying the retry's eventual failure. The alternative (non-consuming `&self`) would have allowed multiple retries from one failed run but broken the frozen-pin-at-transition pattern. Codex picked the kernel-discipline-consistent shape.

2. **Lineage on outer `Run<S>`**: same shape Brief 13 established for intent/claims. Generic `impl<S: RunState> Run<S>` provides `retried_from()` / `retry_index()` getters uniformly; the existing state-specific transitions just copy the new fields along with the existing ones. Codex extended Brief 13's pattern rather than introducing a parallel structure.

3. **`derive_run_id` four-arg signature**: `derive_run_id(intent, claims, retried_from: Option<&str>, retry_index: u32)`. The alternative the Brief flagged was a separate `RunLineage` struct that becomes the second arg. Codex picked the four-arg shape — flatter, less typed-state machinery for a function that already hashes a fixed set of inputs. The Brief had no strong preference; either was fine. The four-arg shape is slightly easier to call when the lineage inputs are already in scope as separate values, which is the dispatch and retry call sites' situation.

4. **u32 overflow handling**: Codex chose `checked_add(1).expect("retry_index exceeded u32::MAX; start a new dispatch instead of retrying")`. This is a defensive Rust idiom: the panic message names the failure mode and the recovery path. The Brief didn't specify this; Codex made a defensible idiom choice. A retry chain of u32::MAX depth is operationally impossible (4+ billion retries would take longer than the lab's existence), so this is exactly the right "this should never happen but if it does, fail clearly" treatment.
